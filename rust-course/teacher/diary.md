# 学习日记

用第一人称、生活化记录每次学习后的感受与收获（重点：真实、具体、可回忆）。

---

- 2026-03-15

  今天终于把 Rust 的环境跑通了。刚开始明明 `rustc -V` 和 `cargo -V` 都能用，我还以为万事大吉，结果一到 `cargo run` 就被 `link.exe not found` 当头一棒，瞬间意识到"能识别命令"和"能真正编译项目"根本不是一回事。后来切到正确的 MSVC 开发环境，再看到终端里那句 `Hello, world!`，心里一下就踏实了。

  接着学了 Cargo 最基础但最实用的几个命令。以前我对这些命令只有模糊印象，现在终于理顺了：`cargo check` 是最快的检查，`cargo build` 是只编译，`cargo run` 是编译加运行，`--release` 会生成优化版本。这样的区别以前我也许会靠死记，现在是真的能从自己的操作经验里分辨出来了。

  后半节转到变量绑定。Rust 默认不可变这件事，刚听上去有点"反常识"，但一旦自己写出那个 `cannot assign twice to immutable variable` 的报错，反而觉得它很直白。`mut` 是给同一个变量加"后面会变"的标记，而遮蔽则是重新创建一个同名的新变量，这个区别今天算是真正进脑子了。尤其是看到遮蔽甚至可以把整数变成字符串、再变成布尔值，这一下让我感觉 Rust 的设计非常有意思：严格，但不死板。

- 2026-03-15

  今天继续往"基本类型"推进，本来以为会是偏记忆的一节，结果还是被 Rust 的细节教育了。最开始我居然把 `parse().expect(...)` 里的报错字符串理解错了，后来才反应过来：编译器卡住根本不是因为那句提示文案，而是 `parse()` 能解析成很多类型，它不知道我到底想要 `i32` 还是别的。这种"不是不会写代码，而是没把类型信息说清楚"的感觉，挺 Rust。

  另外一个今天反复打架的点，是 `"A"` 和 `'A'`。看着都像一个字母，但前者是 `&str`，后者才是 `char`。以前在别的语言里我可能根本不会认真区分到这一步，但 Rust 会逼着我把"一个字符"和"一段字符串"分开想。还有 `char` 能装 Unicode，所以 `'中'`、`'😻'` 都成立，这个细节让我对它为什么占 4 字节一下就有感觉了。

  最后真正让我卡住的是"表达式和语句"。我能隐约感觉分号有影响，但一开始总把关系说反。现在至少顺过来了：代码块最后一行如果没有分号，它就在返回一个值；一旦加了分号，往往就只剩下 `()`。`fn f1() -> i32 { 3 }` 和 `fn f2() -> i32 { 3; }` 这种对比特别有杀伤力，能一下把这个规则钉住。

- 2026-03-15

  今天把数值类型这一节也顺下来了。刚开始感觉像是在背表格：`i8/i16/i32`、`u8/u16/u32`、`isize/usize`，但聊着聊着发现关键不是背名字，而是背后那套语义。比如 `usize` 不是"一个神秘的大整数"，它本质上是和平台指针宽度绑定的，所以数组下标、长度、容量这些值常用它就很合理。

  我今天还第一次认真区分了 `'A'` 和 `b'A'`。前者是 `char`，后者是 `u8`，一个是字符，一个是字节。看上去只差一个 `b`，语义却完全不同。Rust 这种地方很烦，但也很诚实：它不允许我糊里糊涂地把"长得像"当成"就是一样"。

  浮点数那部分也挺有冲击力。以前总觉得 `0.1 + 0.2 == 0.3` 这种事还需要提醒吗，结果 Rust 直接让我重新尊重"近似值"这件事。还有 `NaN` 连和自己都不相等，确实有点反直觉。最后的范围和位运算算是开了个头：`1..5` 不包含右端点，`1..=5` 才包含；位移如果超出位数还会直接报错。感觉 Rust 一直在重复同一个性格：边界必须说清楚。

- 2026-03-16

  今天终于进入了传说中的所有权章节。说实话，在学之前我就听说这是 Rust 最"反人类"的部分，但真正聊下来，感觉更像是"反直觉但合理"。

  小七先带我梳理了内存管理的三种流派：GC、手动、所有权。GC 自动但有性能损耗和 STW 卡顿；手动性能极致但容易出错（悬空指针、内存泄漏、二次释放）。所有权则是第三条路：编译期检查，运行时零成本。这个思路一下就让我理解了为什么 Rust 敢说自己"安全且高性能"——它不是运行时做检查，而是编译时就帮你排雷。

  所有权的三条规则其实不难记：每个值有一个所有者；一个值只能有一个所有者；所有者离开作用域时值被 drop。真正有意思的是第二条规则的推论——如果我写了 `let s2 = s1`，而 `s1` 是个 `String`，那 `s1` 马上就会失效。这不是 bug，是设计。因为堆上的数据如果允许两个所有者，离开作用域时就会"二次释放"，造成内存污染。

  我第一次真正感受到 Rust 的"严格"是在看错误信息的时候。`error[E0382]: borrow of moved value` 这行报错，Rust 不光告诉你错了，还告诉你 "move occurs because `s1` has type `String`, which does not implement the `Copy` trait"，以及 "value moved here"、"value borrowed here after"。这种级别的提示，让我感觉编译器不是在骂我，而是在教我。

  Copy 和 Move 的区别今天算是真正懂了：栈上的简单类型（i32、bool、char）自动 Copy，赋值后旧变量还能用；堆上的复杂类型（String、Vec）是 Move，赋值后旧变量失效。元组只有当所有元素都 Copy 时才是 Copy，`(i32, String)` 赋值会 Move，这个细节很有意思。

  最后学到借用（`&`）的时候，感觉豁然开朗。原来我不想转移所有权，只是想临时用一下的时候，可以用 `&s` 来"借"。这让我想起了 C++ 的引用，但 Rust 的借用检查器会在编译期确保你不会 dangling、不会数据竞争。借用不转移所有权，用完就还——这个概念比我想象的要自然。

  不过我也意识到，所有权只是入门，借用和引用的细节、生命周期，应该才是真正的难关。下节课要继续加油了。

- 2026-03-16

  今天深入学习了借用与引用，这是 Rust 所有权系统的第二个关键部分。

  首先是解引用 `*y`，这让我意识到引用 `&i32` 和值 `i32` 是完全不同的类型，必须用 `*` 才能拿到里面的值。这个细节很基础，但如果不注意，写代码时容易懵。

  然后是可变引用 `&mut T`，我终于理解为什么 Rust 要限制"同一作用域只能有一个可变引用"——这是为了防止数据竞争。想象一下两个人同时修改同一份文档，结果肯定乱套。Rust 在编译期就禁止这种情况，而不是等到运行时崩溃。这种"宁可编译不过，也不要运行时 bug"的态度，我现在有点喜欢了。

  不可变引用和可变引用不能共存这条规则，一开始觉得好严格。但仔细想：如果我正在读一份文件（不可变引用），突然有人把它改了（可变引用），那我读到的数据就不一致了。Rust 的解决方案是干脆不让这两种情况同时发生。

  NLL（Non-Lexical Lifetimes）是个惊喜。原以为引用作用域一定要到花括号结束，结果发现现代 Rust 编译器会看"最后一次使用"在哪里。这让我们在写代码时不用那么小心翼翼，编译器变得更聪明了。

  最震撼我的是悬垂引用的例子。写了个函数返回局部变量的引用，编译器直接报错说 "missing lifetime specifier"。一开始没懂，后来才明白：局部变量函数结束就 drop 了，返回的引用指向了已经被释放的内存！在其他语言里这可能就是运行时崩溃，Rust 直接不让你编译通过。

  今天最大的感悟是：Rust 的"难"不是故意刁难，而是逼你在写代码时就考虑清楚内存安全。这些限制一旦理解了背后的原因，反而觉得很自然。下节课学字符串和切片，应该会有更多实战。

- 2026-03-18

  今天终于碰到了 Rust 里那个大家都说很容易翻车的点：字符串。果然名不虚传。刚开始我还是下意识把 `String` 和 `&str` 当成“都差不多是字符串”，结果一进函数参数就被打脸了：`"Pascal"` 这种字面量其实是 `&str`，而不是 `String`。这一下让我更能体会 Rust 的脾气了——表面上看差不多的东西，只要所有权语义不同，它就一定要你分清楚。

  今天最重要的收获，是开始真的把“字符串切片是按字节，不是按字符”记进脑子里。之前看到 `&s[..5]` 这种写法，总会下意识把它理解成“前五个字符”，但 Rust 根本不是这么想的。尤其一碰到中文，`&s[0..2]` 这种写法直接 panic，冲击感特别强。现在至少知道了：UTF-8 下一个字符可能占 1 到 4 个字节，所以切片必须卡在合法边界上。

  另外我还开始区分字符串操作到底是“改原串”还是“返回新串”。这个点以前我很容易混，比如直觉上就会把 `replace()` 当成原地修改，结果 Rust 偏偏不是，它会返回一个新的 `String`；反过来 `push`、`clear`、`remove`、`replace_range` 这些才是真的在原字符串上动刀。感觉 Rust 经常逼我把“这一步到底有没有发生所有权、借用、原地修改”想得特别具体。

  `pop()` 返回 `Option<char>` 这件事也挺有 Rust 味的。它不愿意假装“肯定有字符可弹”，而是直接把“可能没有”写进类型里。现在我还没完全习惯这种思路，但已经能感觉到这比运行时默默出错靠谱得多。总的来说，今天这节虽然细碎，但我第一次觉得自己开始真正摸到 Rust 字符串系统的门了。

- 2026-03-19

  今天把字符串这一章后半段也收掉了，感觉终于不是“会背几个 API 名字”，而是真的开始知道什么时候该用哪一种方式。比如字符串拼接，`+` 和 `format!` 看上去都能得到新字符串，但语义完全不一样：`+` 会把左边那个 `String` move 掉，而 `format!` 更像是重新组织出一个新的结果，不会把原来的变量顺手拿走。这个细节挺像 Rust 的一贯风格——表面上功能相近，底层语义还是得说清楚。

  原始字符串也让我眼前一亮。像路径、引号、反斜杠这些东西，以前一多起来就开始满屏 `\\`、`\"`，看着就头大。现在知道了 `r"..."` 和 `r#"..."#` 这种写法，至少能把“我到底想表达什么文本”看清楚，不会陷在转义符里打架。

  另一个收获是 `chars()` 和 `bytes()` 的区别终于顺下来了：前者按字符看，后者按字节看。这个区分其实正好把整章的主线又收了一遍——Rust 字符串底层是 UTF-8 字节序列，但人写业务时经常关心的是字符。什么时候该按字节思考，什么时候该按字符思考，今天算是第一次有点分层的感觉了。

  现在回头看，字符串这一章虽然很碎、坑也多，但它把所有权、借用、UTF-8、标准库 API 的思维方式全拧到一起了。能学明白这章，感觉自己对 Rust 的“别糊弄，边界说清楚”这套风格又熟了一层。下一节是元组，应该会轻一点，但也正好让我喘口气。

- 2026-03-24

  今天学了元组，果然比字符串那章轻一点，但也不是“看一眼就完事”的那种简单。最先钉住的一点，是元组和数组虽然都像“装多个值”，但数组要求同类型，元组却可以把不同类型绑在一起。这个差别一说出来很朴素，但它一下就把我脑子里两者的定位分开了：数组更像同类数据集合，元组更像临时打包。

  我今天还犯了一个很典型的小错：明明已经知道元组按位置访问，还是把第二个元素写成了 `.2`。这种错不复杂，但很说明问题——Rust 的这些位置语法不能只靠“我大概懂”，还得真的把从 0 开始这件事练成本能。

  另外一个挺有收获的点，是 `_` 的理解。以前我容易把它想成某种“匿名变量”，但今天更清楚了：它其实是在模式里明确表示“这个位置我不要”。像 `let (_, y, _) = tup;` 这种写法，就很像是我在主动告诉编译器“我只关心中间这个”。

  最有 Rust 味的地方还是所有权。元组本身看着很简单，但只要里面装了 `String`，解构或者取字段时就会发生 move，原元组也可能不能再整体使用。这让我感觉 Rust 很有意思：它不会因为某个语法看上去“轻量”就放松规则。元组这一节像是复习了所有权，也顺手给下节结构体做了铺垫。

- 2026-03-31

  今天把结构体这一节顺下来了，感觉像是把“怎么把数据说清楚”这件事又往前推进了一步。元组虽然省事，但总归还是靠位置记含义；结构体一上来把 `username`、`email` 这种字段名摆出来，代码一下就更像在说人话了。我第一次明显感受到，Rust 不是只在乎“能不能装下这些值”，还很在乎“这些值在业务上到底分别是谁”。

  今天我也踩了几个很典型的坑。比如一开始下意识觉得“改的是字段，所以字段自己可变就行”，后来才真正记住：Rust 管的是整个绑定，想改 `user1.email`，就得让 `user1` 本身是 `mut`。还有结构体初始化时，我也会顺手把 `count: 0` 写成 `count = 0`，这种小错误很像刚从别的语言切过来的惯性。

  结构体更新语法 `..old` 这块挺有意思，也挺容易嘴上说对、脑子里却没真想清。今天我终于把那句“不是所有字段都 move”记牢了：像 `u32`、`bool` 这种 `Copy` 字段只是拷贝，真正会让旧结构体失去完整可用性的，是 `String` 这种非 `Copy` 字段。还有一个细节也顺便纠正了：字段被 move 走，不等于它已经立刻 drop，而是所有权已经转移，原结构体因此不能再整体拿来用。

  最后补上的元组结构体和单元结构体，也算把这章收圆了。`struct Point(i32, i32, i32);` 这种以前看着像元组，现在知道它其实是“有类型名但字段无名”；而 `struct Marker;` 则更极端，连字段都没有，只保留一种类型的身份。感觉今天这节不算难，但特别适合把“语法、语义、所有权”三件事重新拧在一起。

- 2026-04-17

  今天开始进入流程控制，这节里最让我重新建立直觉的点，是 Rust 的 `if` 居然不只是“执行分支”的语句，还能直接当表达式产出一个值。像 `let number = if condition { 5 } else { 6 };` 这种写法，一开始我其实还是会下意识把 `if` 当成“做判断的语法壳子”，但今天被反复追问之后，终于更清楚了：`if` 本身就能返回值，所以才能直接放在赋值号右边。也正因为它是在“产出一个值”，两个分支的类型就必须一致，`5` 和 `"six"` 这种混搭是绝对不行的。

  循环这块今天也把几个之前容易混的点拆开了。`for item in collection`、`for item in &collection`、`for item in &mut collection`，表面上都像是在“遍历集合”，但背后的所有权语义完全不同。尤其是数组里装 `i32` 和数组里装 `String` 的对比，让我更有感觉：是不是还能在循环后继续使用原集合，不是看“它长得像不像数组”，而是要看元素类型是不是 `Copy`。这种地方特别 Rust——语法看着很像，真正决定行为的是类型和所有权。

  另外今天还顺手把 `continue`、`break`、`while` 这些基础控制流重新钉了一遍。`continue` 是跳过当前这一轮，`break` 是直接结束整个循环；`while` 则更像是“条件成立就一直干”。这节内容不算难，但我感觉它把前面学过的“表达式 vs 语句”“mut 可变性”“Copy / Move / 借用”全都串起来了。现在再看流程控制，已经不是只会写 if/for，而是会开始想：这段代码到底是在产出值，还是在做动作；是在借用数据，还是把所有权拿走。

- 2026-04-16

  今天学了数组，这一节表面上不长，但它把 Rust “类型要说清楚” 这件事又往前推了一步。以前看到 `[i32; 5]` 这种写法，我容易把它模糊地理解成“一个整型数组”，今天才真正把里面两个维度分开：`i32` 说的是元素类型，`5` 说的是数组长度，而且长度不是附属信息，而是类型本身的一部分。所以 `[i32; 3]` 和 `[i32; 4]` 虽然看起来很像，其实根本不是同一种类型。

  另一个今天终于顺下来的点，是我开始把“类型写法”和“值写法”分开了。`[u8; 4]` 是类型，`[1; 4]` 是值，后者表示把 `1` 重复四次，展开就是 `[1, 1, 1, 1]`。这个地方我一开始还会困惑“为什么 3 也能写进方括号里”，现在明白了：那不是类型，而是构造数组值的简写语法。Rust 真的很爱用相似语法考验我有没有把语义分清。

  今天还有一个很有 Rust 味的地方，是数组下标越界不会“凑合着给个结果”，而是直接 panic。像 `a[3]` 这种访问，一旦超出范围，Rust 宁可让程序停下来，也不允许我像在某些低层语言里那样越界乱读。现在我对数组和 `Vec<T>` 的区别也终于能用一句话说清了：数组是编译期长度固定的一段同类型数据，`Vec<T>` 才是可以动态变化长度的可增长容器。感觉这节虽然基础，但把我对“类型、长度、边界”三件事的直觉又钉牢了一些。

- 2026-04-08

  今天学了枚举，这节最重要的收获不是“记住 enum 语法”，而是终于开始把结构体和枚举的建模边界分开了。结构体适合表达“一个值同时有这些字段”，枚举适合表达“一个值只能是几种情况之一”。这句话以前我可能能背出来，但今天在 `Message`、`PokerCard`、`Option<T>` 这些例子里反复追问之后，感觉才算真正进脑子。

  我今天暴露出来的一个典型问题，是容易把“变量的类型”和“变体内部携带的数据”混成一件事。看到 `Message::Write(String::from("hi"))`，我下意识会想“这里不是 String 吗”，但现在更清楚了：里面装的是 `String`，整个值的类型仍然是 `Message`。这个点很 Rust，也很关键，因为它决定了我到底是在处理“一个普通字符串”，还是在处理“一条消息中的某个分支”。

  `Option<T>` 这部分也让我感觉 Rust 的味道越来越明显了。以前在别的语言里，空值更像是一个偷偷混进来的特殊状态；但 Rust 直接把“有值/没值”做成了类型的一部分，逼我先面对 `Some` 和 `None` 两个分支，才能继续往下写。现在我对“为什么说 `Option<T>` 比 null 更安全”终于不是停留在口号层面，而是能从“`Option<T>` 和 `T` 根本不是一种类型”这个角度去理解了。
